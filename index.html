✅ Steps to Migrate to OpenFGA for Authorization

1️⃣ Keep Azure AD for Authentication

Azure AD will still be your identity provider (IdP) to handle login, user authentication, and issuing access tokens.

2️⃣ Model Your Authorization in OpenFGA

OpenFGA uses Relationship-Based Access Control (ReBAC) instead of RBAC.

You'll need to define your authorization model using Tuples (actor-action-object relationships).

Example:

Instead of roles, OpenFGA allows you to express relationships:

user:123 has role admin in org:456
    user:789 is a member of group:123

3️⃣ Define Authorization Model in OpenFGA

Here’s an example of an OpenFGA authorization model:

{
      "schema_version": "1.1",
      "type_definitions": [
        {
          "type": "user"
        },
        {
          "type": "organization",
          "relations": {
            "admin": {
              "this": {}
            },
            "member": {
              "this": {},
              "union": { "admin": {} }
            }
          }
        },
        {
          "type": "document",
          "relations": {
            "owner": { "this": {} },
            "editor": { "this": {}, "union": { "owner": {} } },
            "viewer": { "this": {}, "union": { "editor": {} } }
          }
        }
      ]
    }
  • Users have roles in an organization (admin, member)
  • Documents have owner, editor, and viewer roles

4️⃣ Store Authorization Data as Tuples

Once the model is defined, store relationships (tuples) in OpenFGA:

[
      { "user": "user:123", "relation": "admin", "object": "organization:456" },
      { "user": "user:789", "relation": "member", "object": "organization:456" },
      { "user": "user:123", "relation": "owner", "object": "document:abc" }
    ]

5️⃣ Modify Your App to Use OpenFGA for Authorization

Instead of checking roles via Keycloak, update your API or microservices to call OpenFGA.

Example API call to check if user:123 can edit document:abc:

http
    CopyEdit
    POST /check
    {
      "user": "user:123",
      "relation": "editor",
      "object": "document:abc"
    }

If OpenFGA returns true, grant access. Otherwise, deny it.

6️⃣ Migrate Existing Roles & Permissions

  • Extract existing role assignments from Keycloak.
  • Convert roles into OpenFGA relationships (tuples).
  • Populate OpenFGA with these relationships.

7️⃣ Update Services to Query OpenFGA

  • Modify APIs that check user roles (isAdmin(user)) to call OpenFGA instead.
  • Replace Keycloak role-based permission checks with OpenFGA relationship-based checks.

8️⃣ Testing & Deployment

  • Test OpenFGA responses with different user scenarios.
  • Deploy OpenFGA alongside Keycloak, then gradually switch API authorization checks.

🎯 Why Migrate to OpenFGA?

✅ More Flexible than RBAC – Supports complex access rules (e.g., “edit if you’re a team lead AND assigned to this project”).

✅ Designed for Scale – Handles millions of permissions efficiently.

✅ Fine-Grained Control – Works well for dynamic permissions (e.g., per document, per organization).

CURL with OpenFGA

export FGA_API_URL=http://localhost:8080 # optional. Defaults to http://localhost:8080
    export FGA_STORE_ID=01JKFSJNYCH8BPVPXN0N9PQ34Z # required for all calls except \`store create\`, \`store list\` and \`model validate\`
    export FGA_MODEL_ID=01JKFT14Y6KGN1GZVHM89ZA0BY # optional, can be overridden per request, latest is used if this is empty
    export FGA_API_TOKEN=YOUR_API_TOKEN
    
    export FGA_API_AUDIENCE=YOUR_API_AUDIENCE
    export FGA_CLIENT_ID=YOUR_CLIENT_ID
    export FGA_CLIENT_SECRET=YOUR_CLIENT_SECRET

Config model

DSL
    
    model
      schema 1.1
    
    type user
    
    type document
      relations
        define reader: [user]
        define writer: [user]
        define owner: [user]
        
        
    fga model write --store-id=${FGA_STORE_ID} --format=json '{"schema_version":"1.1","type_definitions":[{"type":"user"},{"type":"document","relations":{"reader":{"this":{}},"writer":{"this":{}},"owner":{"this":{}}},"metadata":{"relations":{"reader":{"directly_related_user_types":[{"type":"user"}]},"writer":{"directly_related_user_types":[{"type":"user"}]},"owner":{"directly_related_user_types":[{"type":"user"}]}}}}]}'

The API will then return the authorization model ID.

The OpenFGA API only accepts an authorization model in the API's JSON syntax. To convert between the API Syntax and the friendly DSL, you can use the FGA CLI.

Perform check

curl -X POST $FGA_API_URL/stores/$FGA_STORE_ID/batch-check \
    -H "Authorization: Bearer $FGA_API_TOKEN" \
    -H "content-type: application/json" \
    -d '{"authorization_model_id":"01JKFT14Y6KGN1GZVHM89ZA0BY","checks":[{"tuple_key":{"user":"user:anne","relation":"writer","object":"document:Z"},"correlation_id":"886224f6-04ae-4b13-bd8e-559c7d3754e1"},{"tuple_key":{"user":"user:anne","relation":"reader","object":"document:Z"},"correlation_id":"da452239-a4e0-4791-b5d1-fb3d451ac078"}]}'
