‚úÖ Steps to Migrate to OpenFGA for Authorization

1Ô∏è‚É£ Keep Azure AD for Authentication

Azure AD will still be your identity provider (IdP) to handle login, user authentication, and issuing access tokens.

2Ô∏è‚É£ Model Your Authorization in OpenFGA

OpenFGA uses Relationship-Based Access Control (ReBAC) instead of RBAC.

You'll need to define your authorization model using Tuples (actor-action-object relationships).

Example:

Instead of roles, OpenFGA allows you to express relationships:

user:123 has role admin in tenant:456
user:789 is a member of group:123

3Ô∏è‚É£ Define Authorization Model in OpenFGA

Here‚Äôs an example of an OpenFGA authorization model:

{
  "schema_version": "1.1",
  "type_definitions": [
    {
      "type": "user"
    },
    {
      "type": "organization",
      "relations": {
        "admin": {
          "this": {}
        },
        "member": {
          "this": {},
          "union": { "admin": {} }
        }
      }
    },
    {
      "type": "document",
      "relations": {
        "owner": { "this": {} },
        "editor": { "this": {}, "union": { "owner": {} } },
        "viewer": { "this": {}, "union": { "editor": {} } }
      }
    }
  ]
}
  • Users have roles in an organization (admin, member)
  • Documents have owner, editor, and viewer roles

4Ô∏è‚É£ Store Authorization Data as Tuples

Once the model is defined, store relationships (tuples) in OpenFGA:

[
  { "user": "user:123", "relation": "admin", "object": "organization:456" },
  { "user": "user:789", "relation": "member", "object": "organization:456" },
  { "user": "user:123", "relation": "owner", "object": "document:abc" }
]

5Ô∏è‚É£ Modify Your App to Use OpenFGA for Authorization

Instead of checking roles via Keycloak, update your API or microservices to call OpenFGA.

Example API call to check if user:123 can edit document:abc:

POST /check
{
  "user": "user:123",
  "relation": "editor",
  "object": "document:abc"
}

If OpenFGA returns true, grant access. Otherwise, deny it.

6Ô∏è‚É£ Migrate Existing Roles & Permissions

  • Extract existing role assignments from Keycloak.
  • Convert roles into OpenFGA relationships (tuples).
  • Populate OpenFGA with these relationships.

7Ô∏è‚É£ Update Services to Query OpenFGA

  • Modify APIs that check user roles (isAdmin(user)) to call OpenFGA instead.
  • Replace Keycloak role-based permission checks with OpenFGA relationship-based checks.

8Ô∏è‚É£ Testing & Deployment

  • Test OpenFGA responses with different user scenarios.
  • Deploy OpenFGA alongside Keycloak, then gradually switch API authorization checks.

    define update: [user, role#member] or update from parent

    define view: [user, role#member] or view from parent

@Path("/")
@RegisterRestClient
public interface OpenFGAService {
    @POST
    @Path("/stores/{store_id}/check")
    Uni<Boolean> checkPermission(
            @HeaderParam(HttpHeaders.AUTHORIZATION) String token,
            @PathParam("store_id") String storeId
    );

    @POST
    @Path("/stores/{store_id}/check")
    Uni<Boolean> queryPermission(
            @HeaderParam(HttpHeaders.AUTHORIZATION) String token,
            @PathParam("store_id") String storeId
    );

    @POST
    @Path("/stores/{store_id}/write")
    Uni<Boolean> setPermission(
            @HeaderParam(HttpHeaders.AUTHORIZATION) String token,
            @PathParam("store_id") String storeId
    );
}
